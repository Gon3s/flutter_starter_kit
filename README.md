# Flutter Starter Kit

[![style: very good analysis][very_good_analysis_badge]][very_good_analysis_link]
[![License: MIT][license_badge]][license_link]

A **production-ready Flutter starter kit** built around a clean, feature-first
architecture. It ships with Firebase Authentication (Email/Password), push
notifications, secure session persistence, routing with guards, French/English
localization, and full CI — so you can skip the boilerplate and start building
features on day one.

It can be used in three ways:

1. **As a template** — clone it and start coding.
2. **As a [Mason][mason_link] brick** — generate a fresh project with an
   interactive prompt (Firebase and auth are optional). See [Mason bricks](#-mason-bricks).
3. **As a feature brick** — drop the whole authentication feature into an
   existing Riverpod project.

## ✨ Features

| Area | What you get |
|------|--------------|
| **State management** | [Riverpod 3][riverpod_link] with code generation (`@riverpod`) |
| **Routing** | [GoRouter][go_router_link] with code-gen, auth redirect guards, and a 404 screen |
| **Immutable models** | [Freezed 3][freezed_link] + `json_serializable` |
| **Authentication** | Firebase Auth Email/Password: sign in, sign up, sign out, change password, delete account |
| **Account management** | Dedicated account screen (user info, change password, danger zone) |
| **Push notifications** | Firebase Messaging with foreground/background handlers and FCM token registration |
| **Session persistence** | `flutter_secure_storage`-backed session store with app-startup restoration |
| **Crash reporting** | Firebase Crashlytics wired into the bootstrap zone |
| **Analytics** | Firebase Analytics |
| **Localization** | Lightweight FR/EN string system that follows the device locale |
| **Multi-environment** | `development` / `production` flavors via separate entry points |
| **Error handling** | Typed `AppException` hierarchy + `AsyncValue` UI extensions |
| **Quality** | `very_good_analysis` lints, unit/widget tests, GitHub Actions CI |

## 🏗️ Architecture

The project follows a **feature-first** structure, where each feature is split
into `data`, `domain`, and `presentation` layers (inspired by the
[Riverpod architecture][riverpod_arch_link]).

```
lib/
├── main.dart                 # Production entry point
├── main_dev.dart             # Development entry point
├── bootstrap.dart            # App init: Firebase, Crashlytics, error zone, runApp
├── app.dart                  # MaterialApp.router, theme, localization, FCM setup
├── app_env.dart              # Environment enum + Firebase options selector
├── firebase_options.dart     # Generated by `flutterfire configure` (placeholder committed)
│
├── common_widgets/           # Reusable widgets (PrimaryButton, EmptyPlaceholder…)
├── constants/                # App-wide constants (sizes, spacing)
├── db/                       # Secure storage provider
├── exceptions/               # Typed AppException + AsyncValue UI extensions
├── localization/             # AppStrings (FR/EN) + .hardcoded helper
├── routing/                  # GoRouter config, app startup, refresh stream, 404
├── utils/                    # Notifications, validators, logging, delay helpers
│
└── features/
    ├── authentication/
    │   ├── data/             # AuthRepository impls (Firebase + Fake), session storage
    │   ├── domain/           # AppUser model, AuthRepository contract
    │   └── presentation/
    │       ├── sign_in/      # Sign-in screen + controller
    │       ├── sign_up/      # Sign-up screen + controller
    │       └── account/      # Account screen + controllers
    └── home/
        └── presentation/     # Home screen + counter controller
```

### Key patterns

- **`AuthRepository` abstraction** — the app depends on the `AuthRepository`
  interface, not on Firebase directly. `FirebaseAuthRepository` is the real
  implementation; `FakeAuthRepository` is used in tests. Swap the
  implementation in `lib/features/authentication/domain/auth_repository.dart`.
- **App startup gate** — `appStartupProvider` loads the persisted session and
  registers the device before the UI is shown. The router redirects to a startup
  screen while it resolves and to an error screen on failure.
- **Auth redirect guards** — `goRouter` watches the auth state and redirects
  logged-out users to `/login` and logged-in users away from auth screens.

## 📦 Requirements

- **Flutter** ≥ 3.10 (stable channel) — the project targets Dart `>=3.8.0 <4.0.0`,
  but some dependencies (e.g. `go_router 17`) require **Dart 3.10+**, so use a
  recent stable Flutter. Run `flutter upgrade` if `pub get` reports a version
  conflict.
- **A Firebase project** (only if you keep Firebase / auth enabled).
- **[FlutterFire CLI][flutterfire_link]** to generate platform credentials.
- **[Mason CLI][mason_cli_link]** (optional) if you want to generate projects
  from the bricks.

## 🚀 Getting Started

### 1. Clone the repository

```bash
git clone https://github.com/gon3s/flutter_starter_kit.git
cd flutter_starter_kit
```

### 2. Install dependencies

```bash
flutter pub get
```

### 3. Configure Firebase

This repo **does not** contain real Firebase credentials — only a committed
placeholder `lib/firebase_options.dart` so code generation can run in CI.
Generate your own with the FlutterFire CLI:

```bash
# Install the CLIs once
dart pub global activate flutterfire_cli
npm install -g firebase-tools
firebase login

# Generate lib/firebase_options.dart for your project
flutterfire configure
```

`flutterfire configure` also creates the native credential files:

- `android/app/google-services.json`
- `ios/Runner/GoogleService-Info.plist`

> ⚠️ These files contain project keys. Keep them out of version control if your
> repo is public (add them to `.gitignore`).

In the Firebase console, enable **Authentication → Sign-in method →
Email/Password**.

### 4. Generate code

The project uses code generation for Riverpod, GoRouter, Freezed, and JSON
serialization. Run the builder after any change to annotated files:

```bash
dart run build_runner build --delete-conflicting-outputs
# or, while developing:
dart run build_runner watch --delete-conflicting-outputs
```

### 5. Run the app

The project has two flavors, each with its own entry point:

```bash
# Development
flutter run -t lib/main_dev.dart

# Production
flutter run -t lib/main.dart
```

The environment is exposed via `EnvInfo.environment` / `EnvInfo.isProduction`
(see `lib/app_env.dart`).

## 🧪 Testing

```bash
flutter test
```

Tests use [`mocktail`][mocktail_link] and the `FakeAuthRepository` so they run
without a live Firebase backend. Controllers are tested by listening to their
`AsyncValue` state transitions through a `ProviderContainer`.


## 🌍 Localization

Localization is intentionally lightweight (no ARB/intl build step). Two string
classes — `AppStringsFr` and `AppStringsEn` — implement the abstract
`AppStrings` contract. Use them anywhere with a `BuildContext`:

```dart
final strings = AppStrings.of(context);
Text(strings.signIn);
```

`AppStrings.of(context)` reads the device locale (`fr` → French, otherwise
English). Supported locales are declared in `lib/app.dart`. To add a language,
create a new `AppStrings` implementation and register its locale.

## 🧱 Mason Bricks

This repo doubles as a [Mason][mason_link] brick generator. Install Mason:

```bash
dart pub global activate mason_cli
```

### Full app brick (`brick/`)

Generates a complete starter project. Firebase **and** authentication are
optional, controlled by interactive prompts.

```bash
mason add flutter_starter_kit --path ./brick
mason make flutter_starter_kit
```

| Variable | Type | Default | Description |
|----------|------|---------|-------------|
| `app_name` | string | `my_app` | App name (snake_case) |
| `org_name` | string | `com.example` | Organisation identifier |
| `app_description` | string | `A new Flutter app.` | Short description |
| `avec_firebase` | boolean | `true` | Include Firebase (Analytics, Crashlytics, Messaging) |
| `avec_auth` | boolean | `true` | Include authentication (requires Firebase) |

> If `avec_auth` is `true`, `avec_firebase` must also be `true` (enforced by a
> `pre_gen` hook). A `post_gen` hook runs `flutter pub get` + code generation and
> removes unused directories.

### Auth feature brick (`bricks/flutter_auth_feature/`)

Adds the full Firebase Auth feature to an **existing** Riverpod project.

```bash
mason add flutter_auth_feature --path ./bricks/flutter_auth_feature
mason make flutter_auth_feature
```

| Variable | Type | Default | Description |
|----------|------|---------|-------------|
| `app_name` | string | `my_app` | Your existing package name (snake_case) |
| `output_dir` | string | `lib/features/authentication` | Output directory |

## 🤖 Continuous Integration

GitHub Actions (`.github/workflows/tests.yaml`) runs on every push/PR:

1. Checkout + set up Flutter (stable channel)
2. Inject Firebase credentials from secrets (skipped if absent)
3. `flutter pub get`
4. `dart run build_runner build`
5. `flutter analyze --no-fatal-infos`
6. `flutter test`

Firebase secrets are **optional** — the workflow only writes them when the
secret is set. To enable them, add these repository secrets:

| Secret | Maps to |
|--------|---------|
| `FIREBASE_OPTIONS` | `lib/firebase_options.dart` |
| `GOOGLE_SERVICES_JSON` | `android/app/google-services.json` |
| `GOOGLE_SERVICE_INFO_PLIST` | `ios/Runner/GoogleService-Info.plist` |
